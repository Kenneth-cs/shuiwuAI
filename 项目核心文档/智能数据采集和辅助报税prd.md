## 一、 产品需求文档 (PRD) 细化

### 1. 模块一：智能数据采集页 (Input) - 详细功能需求

#### 1.1 输入端处理

* **对话式语义解析 (NLU)**：
* 需识别：收入类型（专票/普票）、适用税率（1%、3% 或免税）、成本支出项目、从业人数、资产总额。
* *边界处理*：若用户输入模糊（如“这个月赚了点钱”），系统需追问“具体金额是多少？是否含税？”。


* **Excel 模板解析**：
* 基于上传的《会小企 01/02 表》，系统需自动提取：
* **利润表 (会小企02表)**：营业收入、营业成本、利润总额。
* **资产负债表 (会小企01表)**：资产总额（季初/季末余额）。


* *容错机制*：如果用户上传的文件格式略有差异，AI 需具备“模糊对齐”能力。



#### 1.2 数据确认中转站 (Confirmation Layer)

* **功能描述**：AI 提取后，不直接跳转报表，而是展示一个“数据卡片”。
* **必填核心指标**：
1. 本期销售额（区分征税、免税、预缴）。
2. 本期利润总额（用于企业所得税）。
3. 从业人数与资产总额（用于判定“小型微利企业”优惠）。



---

### 2. 模块二：辅助报税导航页 (Output) - 详细功能需求

#### 2.1 报表自动生成逻辑

根据采集的数据，系统需自动执行以下逻辑计算（而非让 AI 盲目心算，由后端逻辑结合 AI 提取值完成）：

* **增值税 (小规模)**：
* 逻辑：若月收入 $\le$ 10万（或季 $\le$ 30万），自动填充至“免税额”行次。
* 计算附加税（城建税、教育费附加、地方教育附加）。


* **企业所得税 (A200000)**：
* 根据“从业人数”和“资产总额”自动勾选“小型微利企业”标志位。
* 计算“实际利润额”并应用 5% 或 25% 的有效税率。


* **文化事业建设费**：
* 仅当用户行业涉及“广告”或“娱乐”时激活，按计费销售额 $\times$ 费率计算。



#### 2.2 视觉引导功能

* **行次锚点**：在模拟报表界面上，点击具体数值，弹出气泡显示：“请将此值填入广东电子税务局申报页面的第 [X] 行”。

## 二、 火山引擎 (Doubao-pro) Prompt 设计

为了保证输出的准确性，建议采用 **JSON 输出模式**，以便程序直接读取并存入 MySQL。

### 1. Prompt 方案：财务数据结构化 (提取层)

**System Prompt:**

```markdown
# Role
你是一个资深的中国税务会计助手，精通广东省小规模纳税人报税规则及《小企业会计准则》。

# Task
请从用户提供的【原始输入】（可能是聊天记录或 Excel 提取的文本）中，提取关键的财务指标。

# Response Format (JSON)
必须输出纯 JSON 格式，包含以下字段：
{
  "revenue": {
    "total": 0.0,
    "taxable_1_percent": 0.0,
    "taxable_3_percent": 0.0,
    "tax_free": 0.0
  },
  "profit_info": {
    "total_profit": 0.0,
    "total_cost": 0.0
  },
  "company_status": {
    "employee_count": 0,
    "asset_total_myr": 0.0, // 单位：万元
    "industry_type": "string"
  }
}

# Constraints
1. 仅提取数值，若信息缺失则设为 null。
2. 资产总额若单位为元，需转换为万元。
3. 如果用户提到“开票收入”，默认视为含税收入，需在后续计算中注意。

```

### 2. Prompt 方案：报表映射 (映射层 - 以 A200000 为例)

**System Prompt:**

```markdown
# Role
你是一个广东税务申报专家，负责将结构化财务数据映射到《居民企业所得税月（季）度预缴纳税申报表（A类）》。

# Input Data
{{structured_data}}

# Logic Reference (根据广东税务局规则)
- 行次1：营业收入
- 行次10：实际利润额（对应利润表利润总额）
- 行次15：小型微利企业减免所得税（需判断：人数<=300, 资产<=5000万, 利润<=300万）

# Output Task
请输出报表填报对照表，格式如下：
{
  "report_name": "A200000",
  "mappings": [
    {"row_id": "1", "name": "营业收入", "value": 50000.00, "guide": "填入官方系统第1行"},
    {"row_id": "10", "name": "实际利润额", "value": 12000.00, "guide": "从利润表自动带入"},
    ...
  ]
}

```

## 三、 技术开发文档（核心环节）

### 1. 核心接口逻辑流程 (Backend Logic)

1. **POST `/api/v1/extract**`:
* 接收 `chat_text` 或 `file_id`。
* 调用火山引擎 OCR（如果是图片/PDF）。
* 调用 `Doubao-pro` 运行“提取层” Prompt。
* **存入 MySQL**: `financial_raw_data` 表。


2. **POST `/api/v1/calculate**`:
* 从数据库取出结构化数据。
* 运行后端固定算法逻辑（由于大模型心算不可靠，税额计算由 Python 代码完成，AI 只负责分类）。
* **存入 MySQL**: `tax_report_snapshots` 表。


3. **GET `/api/v1/report/{id}**`:
* 前端根据 ID 获取 JSON 数据，渲染“辅助报税导航页”。



### 2. MySQL 表结构补充 (针对 Demo 验证)

为了验证准确性，建议增加一个 `audit_log` 字段：

```sql
ALTER TABLE tax_report_snapshots 
ADD COLUMN ai_reasoning TEXT; -- 存储AI给出该映射结果的理由，方便Demo期排查

```

### 3. Demo 期准确性验证策略

* **Case 1 (标准对齐)**：将你提供的 `模版-财务报表报送与信息采集.xlsx` 中的数值手工算出结果。
* **Case 2 (压力测试)**：输入一段杂乱的对话：“上个月卖货4万，退货1千，给员工发工资1万，水电费500，资产大概30万吧。”
* **验证标准**：对比 AI 提取的 `total_profit` 是否等于 `40000 - 1000 - 10000 - 500`。
